# 简愈一人食 — MVP 架构计划

> **版本**：v0.1 MVP  
> **目标用户**：45–65 岁、时间充裕、关注脑力与情绪健康的独立女性  
> **核心理念**：不做卡路里计算，以「结构化营养指标 + 状态反馈算法」沉淀专属抗脑雾、稳情绪菜单库  
> **技术栈**：Streamlit · Pandas · CSV（本地持久化）· Plotly（雷达图）· Pillow（海报拼图）

---

### 1.1 模块职责说明

| 模块 | 职责 |
|------|------|
| `app.py` | 唯一 UI 层；管理 `st.session_state`；调用 `src/` 各模块 |
| `database.py` | 统一 CSV CRUD；首次启动检测空表并写入 seed 数据 |
| `algorithm.py` | 纯函数：单条日志 → Final Weight；批量更新 `menu_weights.csv` |
| `recommendation.py` | 晨间三问 → 过滤/打分 → 返回当日推荐菜单列表 |
| `nutrition.py` | 给定 Menu_ID → 统计 7 大营养类别覆盖比例 |
| `visualization.py` | 营养雷达图渲染为 PNG bytes |
| `export.py` | 合成海报：占位图 + 雷达图 + 可选情绪/精力条 |

### 1.2 为何选用 CSV 而非 SQLite（MVP 决策）

- **单人本地使用**，数据量极小（食材 < 200 行，日志 < 1000 行）。
- **可读可编辑**：用户或开发者可直接用 Excel 查看/补充 seed 数据。
- **零配置启动**：无需建库语句，降低 45–65 岁用户部署门槛。
- 若后续需要多设备同步，可在 v0.2 增加 SQLite 适配层（`database.py` 抽象接口不变）。

## 2. CSV 表字段定义

### 2.1 `ingredients_db.csv` — 食材库

| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| `id` | string | ✓ | 唯一标识，格式 `ING_001` | `ING_001` |
| `name` | string | ✓ | 食材中文名称 | `燕麦` |
| `nutrition_category` | enum | ✓ | 营养类别（见 2.1.1） | `谷物杂粮` |
| `role` | enum | ✓ | 在餐盘中的角色（见 2.1.2） | `主食` |
| `notes` | string | | 可选备注（烹饪提示） | `即食或快煮` |

#### 2.1.1 `nutrition_category` 枚举值（7 类，固定）

```
植物雌激素 | 谷物杂粮 | 蛋白质 | 优质脂肪 | 膳食纤维 | 高钙 | 水果
```

#### 2.1.2 `role` 枚举值（4 类，固定）

```
主食 | 配菜 | 加餐 | 汤
```

**约束规则**：
- `id` 全局唯一。
- 同一 `name` 允许存在不同 `role` 的记录（如「核桃」可同时作为加餐与配菜）。

---

### 2.2 `menu_db.csv` — 菜单主库

| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| `menu_id` | string | ✓ | 唯一标识，格式 `MENU_001` | `MENU_001` |
| `menu_name` | string | ✓ | 菜单显示名称 | `温燕麦碗配蓝莓` |
| `ingredient_ids` | string | ✓ | 逗号分隔的食材 ID 列表 | `ING_001,ING_015,ING_032` |
| `energy_tags` | string | ✓ | 逗号分隔的能量标签（见 2.2.1） | `稳糖,高抗氧化` |
| `meal_type` | enum | ✓ | 适用餐次 | `早餐` |
| `prep_minutes` | int | | 预估准备时间（分钟） | `15` |
| `description` | string | | 一句话描述（UI 展示） | `从容的一人食早餐` |

#### 2.2.1 `energy_tags` 预设标签池

```
抗炎 | 稳糖 | 高抗氧化 | 补脑 | 稳情绪 | 高纤维 | 补钙 | 轻断食友好
```

#### 2.2.2 `meal_type` 枚举值

```
早餐 | 午餐 | 晚餐 | 加餐
```

**约束规则**：
- `ingredient_ids` 中每个 ID 必须在 `ingredients_db.csv` 中存在。
- 一条菜单至少包含 2 个食材 ID。
- `energy_tags` 至少 1 个标签。

---

### 2.3 `log_db.csv` — 用户反馈日志

| 字段名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| `log_id` | string | ✓ | 唯一标识，格式 `LOG_YYYYMMDD_NNN` | `LOG_20250618_001` |
| `date` | date | ✓ | 记录日期 ISO 8601 | `2025-06-18` |
| `menu_id` | string | ✓ | 关联菜单 | `MENU_001` |
| `taste_score` | int | ✓ | 口味评分 1–5 | `4` |
| `operation_score` | int | ✓ | 操作体感评分 1–5（从生鲜开始的从容度） | `5` |
| `mood_score` | int | ✓ | 情绪评分 1–5 | `4` |
| `energy_score` | int | ✓ | 精力评分 1–5 | `4` |
| `is_favorited` | bool | ✓ | 是否加入我的菜单库 | `true` |
| `sleep_quality` | enum | | 当日晨间记录的睡眠状态（回溯用） | `良好` |
| `brain_body_load` | enum | | 当日晨间记录的脑力/体力消耗预期 | `中等` |
| `meal_count` | int | | 当日计划餐数 | `3` |

**约束规则**：
- 四个评分字段均为整数，范围 `[1, 5]`。
- `is_favorited` 存储为 `true` / `false` 字符串（Pandas 读写兼容）。
- 同一 `date` + `menu_id` 允许多条记录（如同一菜单早晚餐各评一次），算法取该菜单**最新一条**或**当日均值**（MVP 采用：**当日同一 menu_id 取均值**）。

---

### 2.4 `menu_weights.csv` — 菜单 NPS 权重缓存（衍生表）

> 由 `algorithm.py` 在每次夜间结算后自动重算写入，非用户手动编辑。

| 字段名 | 类型 | 说明 | 示例 |
|--------|------|------|------|
| `menu_id` | string | 关联菜单 | `MENU_001` |
| `base_score` | float | 基础得分 | `0.82` |
| `multiplier` | float | 状态乘数 | `1.1` |
| `final_weight` | float | 最终系统权重 | `0.902` |
| `is_favorited` | bool | 是否曾被收藏（任一日志为 true 即 true） | `true` |
| `log_count` | int | 参与计算的日志条数 | `3` |
| `updated_at` | datetime | 最后更新时间 | `2025-06-18T22:30:00` |

## 3. 核心算法规范

### 3.1 NPS 权重计算（单条日志）

```
基础得分 (Base Score) = (taste_score / 5) × 0.7 + (operation_score / 5) × 0.3

状态均值 = (mood_score + energy_score) / 2

状态乘数 (Multiplier) 映射：
  均值 ≤ 1.5 → 0.8
  均值 ≤ 2.5 → 0.9
  均值 ≤ 3.5 → 1.0
  均值 ≤ 4.5 → 1.1
  均值 >  4.5 → 1.2

最终权重 (Final Weight) = Base Score × Multiplier
```

> **实现细节**：均值落在区间边界时取较高乘数（如 3.5 → 1.1）。MVP 采用**四舍五入到 0.5 步长**后再查表，与用户直觉一致。

### 3.2 菜单级聚合权重

对同一 `menu_id` 的全部历史日志：
1. 每条日志独立计算 Final Weight。
2. 菜单权重 = 所有 Final Weight 的**算术平均**。
3. 若任意一条日志 `is_favorited == true`，则该菜单标记为收藏菜单。

### 3.3 推荐池排序规则

```
推荐列表 = 全部菜单
  → 按晨间三问过滤（见 3.4）
  → 排序：
      1. is_favorited == true 的菜单强制置顶
      2. 同组内按 final_weight 降序
      3. 权重相同则按 prep_minutes 升序（更从容优先）
```

### 3.4 晨间三问 → 推荐过滤逻辑

| 输入项 | 选项 | 过滤/加权规则 |
|--------|------|---------------|
| **昨晚睡眠状态** | 很好 / 良好 / 一般 / 较差 | 较差 → 优先推荐含 `稳情绪` 标签；很好 → 可推荐 `轻断食友好` |
| **今日脑力/体力消耗** | 低 / 中等 / 高 | 高 → 优先 `补脑` + `稳糖`；低 → 优先 `高抗氧化` |
| **今日一人食餐数** | 1 / 2 / 3 | 决定返回菜单数量上限；3 餐时覆盖早/午/晚各至少 1 道 |

**MVP 简化策略**：不做复杂 ML，采用**标签匹配加分**（+0.1 per matched tag）叠加到 `final_weight` 上进行当日排序。

## 4. Streamlit 页面路由设计

### 4.1 路由方案

采用 Streamlit **原生多页应用**（`st.navigation` + `st.Page`，Streamlit ≥ 1.36），侧边栏极简导航，默认隐藏高级设置。

```
app.py
├── Page: 🌅 晨间分配      → pages/morning.py
├── Page: 🌙 夜间结算      → pages/night_checkin.py
└── Page: 📸 一键导出      → pages/export_poster.py
```

**备选方案**（若 Streamlit 版本较低）：在 `app.py` 内用 `st.radio` 切换三个 `render_*()` 函数，逻辑等价。

### 4.2 全局 Session State 键

| Key | 类型 | 用途 |
|-----|------|------|
| `today_date` | date | 当前操作日期 |
| `morning_inputs` | dict | `{sleep, load, meal_count}` |
| `today_recommendations` | list[str] | 当日推荐 menu_id 列表 |
| `selected_menu_id` | str | 夜间结算时用户选择的菜单 |
| `last_log_id` | str | 最近一次写入的日志 ID（供导出页使用） |
